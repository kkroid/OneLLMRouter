# 大任务 Checklist 追踪

将大任务分解为可追踪的 checklist 文档，放在项目根目录下持久化，跨会话保留进度。

---

## 工作模式

### 模式 A：创建新 Checklist

当用户描述一个大任务时：

1. **先理解**：阅读相关代码和文档，确认任务范围和目标
2. **创建 checklist 文件**：写入项目根目录 `<任务名>.md`
3. **汇报计划**：展示 Phase 划分、验收标准、风险，请用户确认
4. **等待确认**：用户确认后才开始执行第一项

### 模式 B：继续已有 Checklist

当用户指定一个已有的 checklist 文件时：

1. **读取文件**：找到 checklist 文件
2. **定位下一个未完成项**：找到第一个 `- [ ]` 项
3. **执行该项**：完成该任务
4. **勾选**：将 `- [ ]` 改为 `- [x]`，更新头部状态/日期
5. **汇报**：报告完成情况 + 下一个待办项 + 任何新发现的风险
6. **循环**：如果用户说"继续"，重复 2-5

## Checklist 格式模板

```markdown
# <任务标题>

日期：YYYY-MM-DD（初版） / YYYY-MM-DD（更新1）
状态：**Phase N <当前阶段状态>**

> **最关键的上下文放在引用块里**：当前阻塞项、上次决策结论、最新发现。

---

## 1. 定位

> <一句话核心问题>

**包含 / 不包含**：
- 包含：...
- 不包含：...

---

## 2. 验收标准（缺一不可）

| # | 标准 | 度量 | 状态 |
|---|------|------|------|
| C1 | <标准描述> | <量化指标> | ⏸ |

---

## 3. 任务清单（执行顺序）

### Phase 0：前置确认

- [ ] **P0.1** <任务描述>

**验收**：<此 Phase 的验证方式>

### Phase 1：<阶段名称>

- [ ] **P1.1** <任务描述>

**验收**：<验证方式>

---

## 4. 风险与已知坑

| 风险 | 应对 |
|------|------|
| <风险描述> | <应对措施> |

---

## 5. 不属于本任务的事项

- 不做 XXX（理由）
```

### 格式要点

- **状态标记**：`✅` 过 / `⏸` 暂缓 / `❌` 失败 / `⚠️` 风险 / `🔬` 验证中
- **任务编号**：`PX.Y` 格式（Phase X，第 Y 项）
- **决策点**：标注需要用户确认的步骤
- **日期更新**：每次修改时在头部日期后追加
- **Scope 标注**：涉及多端的任务标注 `[Go]` / `[Panel]` / `[Proto]`，方便看出影响范围

---

## 执行规则

### 勾选即完成
- `- [ ]` → `- [x]`：任务完成
- 不要提前勾选未完成的任务

### 遇到阻塞
- 标注为 `⚠️ 阻塞` 并在头部引用块中记录
- 跳到下一个不依赖此项的任务继续

### 发现新任务
- 追加到对应 Phase 末尾

### Phase 完成后的强制审核
**每个 Phase 完成后，必须自动触发 /audit 自我审核，通过后才能进入下一 Phase。**
